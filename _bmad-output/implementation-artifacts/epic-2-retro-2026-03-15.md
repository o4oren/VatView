## What Went Well

- **Multi-LLM pipeline is effective.** Claude implementing, Windsurf and Cursor reviewing from different perspectives catches more issues than a single reviewer.
- **Epic 1 retro commitments all followed through.** Accessibility checklist, prop contracts, and token-first implementation were applied consistently.
- **Zero regressions, zero blockers.** Clean execution across all 4 stories.
- **Pre-existing bugs surfaced and fixed.** The Android ghost polygon bug (react-native-maps #5052, #5080, #3783) was an old, annoying issue resolved during the redesign work.
- **Reusable patterns established.** `pointerEvents="box-none"` overlay pattern, TranslucentSurface for floating elements, FadeScreen tab transitions, safe area positioning conventions — all carry forward to future epics.

## Challenges

- **Story 2.4 scope grew during review.** Started as "static positioning of floating elements" but expanded to include Android overlay ghost fix, polyline fix, map background deselection, and error-state handling. The story's File List didn't capture all files touched.
- **Android polygon removal bug required unplanned research.** Verifying it was a real upstream issue (GitHub issues), designing the transparent-toggle workaround, and adding cache eviction was significant unplanned work.
- **Manual device verification bottleneck.** Stories 2.2–2.4 all have unchecked device verification tasks. No automated tests for map rendering.
- **Performance debt discovered post-epic.** ThemeProvider re-render cascade and unmemoized marker generators. ThemeProvider and `getDataStatus` memoization were fixed immediately; generator refactor deferred to Epic 3.

## Key Insights

1. **Fix bugs when you're in the code** — the polygon fix timing was right (already touching overlay rendering code), but scope changes must be tracked.
2. **Code review is the safety net for scope drift** — when the reviewer sees files changed that aren't in the story spec, they flag it. Dev either updates the story or creates a separate tracked item.
3. **Performance refactors ride with functional changes** — don't create separate performance stories; fold refactors into the stories that naturally touch those files.
4. **The story file is the record of truth** — File List and Change Log must reflect what actually changed, not just what was planned.
